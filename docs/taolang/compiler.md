# 编译器实现

TaoLang 编译器（`taoc`）采用多阶段编译流程，以 LLVM 为后端，将 `.tao` 源代码编译为本地可执行文件。编译流程在传统编译器架构的基础上，引入了空间预处理、生命周期分析和钩子编织三个 TaoLang 特有阶段，以充分支持 RSOP 范式中的生命周期空间与事件钩子特性。

---

## 编译流程概览

```
源代码 (.tao)
    |
[1] 词法分析 (Lexical Analysis)        -> Token 流
    |
[2] 语法分析 (Syntax Analysis)         -> 抽象语法树 (AST)
    |
[3] 空间预处理 (Space Preprocessing)   -> 空间定义表 + 修正 AST
    |
[4] 语义分析 (Semantic Analysis)       -> 类型化 AST + 符号表
    |
[5] 生命周期分析 (Lifetime Analysis)   -> 生命周期注解 AST
    |
[6] 钩子编织 (Hook Weaving)            -> 钩子增强 AST
    |
[7] 中间代码生成 (IR Generation)       -> LLVM IR
    |
[8] 优化 (Optimization)               -> 优化后 LLVM IR
    |
[9] 目标代码生成 (Code Generation)     -> 可执行文件
```

其中，阶段 1-2 和 7-9 属于传统编译器的通用环节；阶段 3-6 是 TaoLang 编译器的核心，负责处理生命周期空间、变量三层结构和事件钩子等 RSOP 特有语义。

---

## 前端：源代码到 AST

### 阶段 1：词法分析

词法分析器将源代码文本拆分为 Token 流。除常规的标识符、字面量、运算符外，需要特别处理以下 TaoLang 特有的词法元素：

- **RSOP 关键字**：`space`、`using`、`on`、`when`、`goto`、`with`、`set`、`to`、`of`、`in`、`as`、`del`、`is`、`self`、`global`、`super`、`break` 等
- **三种变量声明关键字**：`let`、`const`、`def`
- **字符串插值语法**：`$"...{expr}..."` 中的表达式边界识别
- **上下文参数语法**：`[self] ->` 中的方括号与箭头组合

每个 Token 均携带源文件位置信息（行号、列号），用于后续阶段的错误报告。

```
// 输入
space auth { let user = null }

// 输出 Token 流
[SPACE, IDENT("auth"), LBRACE, LET, IDENT("user"), ASSIGN, NULL, RBRACE]
```

### 阶段 2：语法分析

语法分析器根据 TaoLang 的语法规则，将 Token 流组织为抽象语法树（AST）。需要识别的核心结构包括：

- **空间定义**：`space name { ... }` 及其绑定的钩子块
- **函数定义**：`fn name(...) -> type { ... }`
- **变量声明**：`let`/`const`/`def` 及可选的钩子绑定 `[self] -> on/when ...`
- **控制流**：`if`/`while`/`for`/`do loop`/`break`/`goto` 等
- **表达式**：`of` 属性查询、`in` 空间限定、`with` 嵌套关系、`as` 委托声明

语法分析阶段不对语义合法性进行判断，仅确保代码符合语法规则。

---

## 中端：TaoLang 核心分析

中端是 TaoLang 编译器区别于传统编译器的关键部分，由空间预处理、语义分析、生命周期分析和钩子编织四个阶段构成。

### 阶段 3：空间预处理

空间预处理是整个编译流程中最具 TaoLang 特色的阶段。它负责将源代码中分散的空间定义汇总为统一的空间定义表，并据此修正 AST 结构。

这一阶段的工作直接对应 TaoLang 的核心设计原则：**空间定义时平级独立，使用时可任意嵌套组合**。

具体步骤：

1. **空间收集**：遍历整个 AST，收集所有 `space` 定义节点，无论其在源代码中出现的位置。
2. **空间扁平化**：将语法上嵌套的空间定义展开为平级结构。例如 `space outer { space inner { ... } }` 中的 `inner` 被提取为独立的顶层空间。
3. **初始化顺序分析**：分析空间之间的依赖关系（如空间 A 的初始化代码引用了空间 B 的变量），确定安全的初始化顺序，并检测循环依赖。
4. **钩子提取**：从每个空间的 AST 节点中提取 `on`/`when` 钩子，建立钩子查找表。
5. **约束验证**：检查 `main of space == null`（主函数必须在 null 层级）、空间名唯一性等编译时约束。
6. **AST 修正**：将展开后的空间定义移至 AST 顶层，保留 `using` 语句的嵌套结构不变。

```
// 源代码中的嵌套写法
space outer {
    let x = "outer"
    space inner { let y = "inner" }
}

// 空间预处理后，AST 中的空间定义变为：
// 顶层空间 1: space outer { let x = "outer" }
// 顶层空间 2: space inner { let y = "inner" }
```

### 阶段 4：语义分析

在空间预处理完成后，语义分析阶段对修正后的 AST 进行类型检查和名称解析：

| 分析项 | 说明 |
|--------|------|
| 符号表构建 | 为每个作用域（函数、空间、块）建立独立的符号表 |
| 名称解析 | 解析标识符引用，处理 `using` 引入的空间变量可见性 |
| 类型推断与检查 | 推断未显式标注的类型，验证类型兼容性 |
| `of` 属性解析 | 验证 `x of type`、`x of space` 等属性查询的合法性 |
| `with` 关系验证 | 确保 `a with b` 中 a、b 均为合法空间引用 |
| 钩子语义检查 | 验证钩子的事件类型与绑定目标匹配（如变量不能有 `on enter` 钩子） |
| 变量遮蔽分析 | 处理嵌套 `using` 中同名变量的遮蔽关系 |

### 阶段 5：生命周期分析

生命周期分析是 TaoLang 支持"变量即生命"理念的技术基础。此阶段追踪每个变量和空间在程序执行路径上的状态转换，并在 AST 中标注需要触发事件的位置。

核心工作：

- **变量生命周期追踪**：识别每个变量的 `create`、`use`、`modify`、`delete`、`free` 事件点在代码中的具体位置。
- **空间生命周期追踪**：识别每个空间的 `create`、`enter`、`exit`、`delete`、`free` 事件点。
- **自动释放判断**：对 `alive[auto] == true` 的变量，分析其最后一次使用位置，确定最佳释放时机。
- **事件点标注**：在 AST 节点上添加生命周期注解，指示后续的钩子编织阶段应在哪些位置插入钩子调用。

```
// 生命周期分析的标注示意
let x = 10          // [标注: create 事件]
println(x)          // [标注: use 事件]
x = 20              // [标注: modify 事件]
                     // [标注: free 事件 -- 此处是 x 的最后使用后位置]
```

### 阶段 6：钩子编织

钩子编织类似面向切面编程（AOP）中的编织过程，将钩子定义转化为实际的函数调用，并插入到 AST 中由生命周期分析标注的位置。

具体工作：

- **钩子代码生成**：为每个 `on`/`when` 钩子生成对应的内部函数。
- **调用点插入**：在 AST 中标注了事件的位置插入钩子调用。`when` 钩子在事件发生前插入（拦截默认行为），`on` 钩子在事件发生后插入（附加逻辑）。
- **控制流处理**：处理 `when` 钩子中的 `break`（跳出空间）、`goto`（跳转空间）等控制流语义。
- **上下文准备**：为钩子函数准备 `self`（绑定实体引用）；对于需要外部输入的钩子（`when create`、`when modify`、`when enter`、空间 `on enter`）额外准备 `it`（事件上下文参数）。

```tao
// 编织前的源代码
let x = 10
    [self] -> on create { println("created") }
println(x)

// 编织后的等价逻辑
let x = 10
__hook_x_on_create(x)     // 插入的 on create 钩子调用
__hook_x_on_use(x)        // 插入的 on use 钩子调用（如果定义了 on use）
println(x)
```

## 后端：IR 生成与代码输出

### 阶段 7：中间代码生成

将钩子增强后的 AST 翻译为 LLVM IR。除常规的函数、控制流、表达式翻译外，需要为 TaoLang 特有结构生成对应的 IR：

| 结构 | IR 实现 |
|------|---------|
| 空间定义 | 全局数据结构，包含空间状态和变量存储 |
| 变量三层结构 | 外延层（符号元数据）、内涵层（钩子表 + 元数据 + 指针）、数值层（实际数据存储） |
| 钩子函数 | 普通函数，接收 `self` 和 `it` 参数 |
| `as` 委托变量 | 求值函数指针，每次访问时调用 |
| `with` 嵌套关系 | 运行时空间栈结构 |
| `of` 属性查询 | 元数据查找调用 |

### 阶段 8：优化

在 LLVM 通用优化之上，TaoLang 编译器执行以下针对性优化：

| 优化项 | 说明 |
|--------|------|
| 内涵层消除 | 没有钩子的变量退化为传统两层结构，消除内涵层开销 |
| 空钩子移除 | 钩子体为空（`on create {}`）时，移除对应的调用点 |
| 钩子内联 | 简单钩子（如仅含一条语句）直接内联到调用点 |
| 生命周期优化 | 提前释放不再使用的变量，延迟初始化尚未使用的变量 |
| 死空间消除 | 移除从未被 `using` 引用的空间及其初始化代码 |
| 常量折叠 | `const` 变量在编译时替换为字面值 |

内涵层消除是最重要的 TaoLang 特有优化，它保证了零成本抽象的承诺：不使用钩子的变量在编译后与传统语言中的普通变量完全一致。

```tao
// 无钩子变量 -- 优化后等同于传统两层变量
let x = 42
println(x)
// IR: alloc isize x = 42; call println(x)

// 有钩子变量 -- 保留三层结构
let y = 42
    [self] -> on use { println("accessed") }
println(y)
// IR: alloc {meta, hooks, *data} y; call hook_on_use; call println(*y.data)
```

### 阶段 9：目标代码生成

通过 LLVM 后端将优化后的 IR 编译为目标平台的机器码：

- 寄存器分配与指令选择
- 链接运行时库（空间管理器、生命周期管理器、钩子调度器、内存管理器）
- 生成程序入口代码：初始化所有空间 -> 调用 `main` -> 清理全局资源
- 支持多目标平台：本地机器码、WebAssembly 等

## 运行时支持

编译器生成的代码依赖运行时库提供的以下组件：

| 运行时组件 | 职责 |
|-----------|------|
| 空间管理器 | 维护空间状态、处理 `using` 进入/退出、管理嵌套栈、响应 `goto`/`break` |
| 生命周期管理器 | 追踪变量生命周期状态、触发生命周期事件、执行自动回收 |
| 钩子调度器 | 调用钩子函数、传递 `self`/`it` 上下文、处理 `when` 钩子的控制流 |
| 元数据系统 | 存储变量和空间的元数据、支持 `of` 查询（类型、符号名、所属空间等） |
| 内存管理器 | 分配/释放内存、维护变量三层结构的内存布局、处理 `free` 的多种语义 |

---

## 编译时与运行时的分工

TaoLang 遵循"偏向编译时而非运行时"的设计原则，尽可能将检查和决策推到编译阶段：

| 特性 | 编译时 | 运行时 |
|------|--------|--------|
| 空间定义收集与扁平化 | 汇总、展开、验证 | -- |
| 空间初始化 | 生成初始化代码和顺序 | 按序执行初始化 |
| 空间 `using` 嵌套 | 验证空间引用合法性 | 动态管理嵌套栈 |
| 类型检查 | 静态类型推断与验证 | -- |
| 生命周期分析 | 确定事件点、标注释放时机 | 触发事件、执行回收 |
| 钩子编织 | 生成钩子代码、插入调用点 | 调用钩子函数 |
| 变量三层结构 | 决定结构层级（二层或三层） | 维护运行时数据 |
| `const` 常量 | 编译期求值、内联替换 | -- |
| `of` 属性查询 | 静态属性编译期解析 | 动态属性查询元数据 |
| `with` 关系 | 静态验证引用合法性 | 动态构建嵌套关系 |

---

## 实现挑战

1. **钩子性能开销的最小化**：确保内涵层消除优化覆盖绝大多数无钩子变量，使零成本抽象落到实处。
2. **自动释放时机的精确判断**：在复杂控制流（条件分支、循环、异常路径）中准确确定 `alive[auto]` 变量的最佳释放点。
3. **空间嵌套栈的高效管理**：`using` 可任意嵌套组合，运行时需高效维护空间栈的进入/退出操作。
4. **三层结构的内存布局优化**：在保留三层语义的前提下，最小化内存占用和访问延迟。
5. **编译时间控制**：生命周期分析和钩子编织引入了额外的编译复杂度，需要在分析精度和编译速度之间取得平衡。
